How an event moves through the platform

Collect, resolve, model, route

One backend handles the full event lifecycle, and every stage is inspectable rather than a step you take on faith. Ten taped events ride the line below — two are malformed, most are duplicates. Press run, then open any station: the line holds while you read the rules it executes.

Collect and validate

station held open · the line pauses while you read the rules

rules · typed schema

Events land through connectors, get normalized by a transform, and are checked against a schema before anything is stored. Malformed payloads bounce out with the exact field that failed — watch the reject tray under gate 01 while the line runs. 

schema ticket.scan · v12
required: [event_id, user_id, occurred_at]
fields:
  user_id:     string     # null → reject
  occurred_at: rfc3339
on_error: reject · field-level detail
02

Resolve identity

station held open · the line pauses while you read the rules

rules · deterministic

Identifiers you define merge events into profiles by rules you control, with priorities that decide contested merges. No model, no probability score — duplicates collapse into the profile your rules select, in the open, and every merge is logged. 

match_rules:                     # deterministic
  - { id: r1, key: email,      priority: 100 }
  - { id: r2, key: device_id,  priority: 60 }
  - { id: r3, key: loyalty_no, priority: 40 }
merge_limit: 5
conflict:    highest_priority_wins
audit:       every merge logged
03

Model against the warehouse

station held open · the line pauses while you read the rules

join · in-place

Warehouse tables map to the same identity graph, so batch history and live signal share one model. Historical data joins in-place — no copy, no extra ETL hop — and profiles pick up a + hist tag as they pass gate 03. 

tables:
  orders:          { join: user_id,    as: order_history }
  loyalty_members: { join: loyalty_no, as: loyalty_state }
mode: in_place    # zero-copy · no extra ETL hop
04

Route and observe

station held open · the line pauses while you read the rules

delivery · logged

Enabled pipelines forward events to destinations you control, with delivery history and queue health queryable end to end. Toggle every destination off above and events queue visibly instead of vanishing — re-enable one and the queue drains. 

destinations:
  - { id: wh_main, type: warehouse, status: enabled }
  - { id: crm,     type: http,      status: enabled, retry: backoff(3) }
  - { id: ads,     type: http,      status: paused }
delivery: at_least_once · history queryable
queue:    visible · drainable
