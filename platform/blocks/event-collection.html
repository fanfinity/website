How event collection works

From every touchpoint
to one normalized stream

Signals arrive messy — different SDKs, different payloads, different clocks. Blocks turns all of it into one typed, validated stream before anything touches your graph.

  1. 01 {% include icon.html name="plug" class="h-5 w-5" %}

    Open connector framework

    Ingest events from web, app, ticketing, streaming, on-site, and offline sources without custom ETL. Raw payloads are normalized and validated as they arrive.

  2. 02 {% include icon.html name="bolt" class="h-5 w-5" %}

    Real-time ingestion

    Events are captured and processed as they happen, not batched overnight. Fan signals reach your identity graph immediately, so recognition and response happen in real time.

  3. 03 {% include icon.html name="check" class="h-5 w-5" %}

    Schema validation

    Every event is validated against a typed schema as it arrives, so graph hygiene is enforced on the way in rather than repaired by nightly cleanup jobs.

  4. 04 {% include icon.html name="stack" class="h-5 w-5" %}

    No parallel databases

    Events flow directly into your Fanfinity warehouse. No vendor silo to sync or reconcile with your sources: one normalized event stream, one resolved graph.

Under the hood

Anatomy of an event,
before and after

A real raw payload from a ticketing app — and what it becomes after the normalizer runs. Press the button and watch each field find its place.

Before · raw payload

{"ev":"scan","uid":"c_8f2",
 "t":1739482621,"src":"app_v3",
 "seat":"A-114","px":"49.5"}

five fields · no types · epoch time · who knows what "px" is

  • ev→event: ticket.scan · typed
  • uid→user_id: c_8f2 · string
  • t→timestamp: 2026-02-13T19:43:41Z · UTC
  • seat, px→properties: seat · price_usd · typed
  • src→source: ticketing/app v3 · connector

After · typed event ✓ schema v2 · validated

event:      ticket.scan
user_id:    c_8f2
timestamp:  2026-02-13T19:43:41Z
source:     ticketing/app v3
properties: { seat: "A-114", price_usd: 49.50 }
schema:     v2 · validated ✓

one typed record · queryable · safe to stitch

Validation happens on arrival — so "repair" is a job nobody has to run.
